# Petrol Pump Serial Queue System - Implementation Plan

## 1. Goal and Scope
Build a mobile-first web app where users can:
- Search petrol pumps
- Take a daily serial number remotely
- Track real-time queue progress
- Confirm arrival by scanning a pump-specific QR code

Build owner tools where pump owners can:
- Register a petrol pump
- Set and update daily serial limits
- View and manage active queue state
- Download and print a unique QR code for check-in

Platform:
- Frontend: Vue 3 (Vite), Tailwind CSS (optional but recommended)
- Backend: Firebase Auth, Firestore, Cloud Functions, Firebase Storage

## 2. MVP Deliverables
- Google sign-in for users and pump owners
- Role-based access (`user`, `pump_owner`)
- Pump registry and searchable pump list
- Daily serial generation per pump (`1..limit`)
- Queue tracking view with:
  - user serial
  - current running serial
  - remaining slots
  - estimated waiting time
- QR scan check-in endpoint and activation flow
- Owner dashboard with queue controls
- Scheduled functions for:
  - auto-expiry of missed check-ins
  - daily reset preparation and status hygiene
- SEO-ready routes for pump pages

- `address: string`
- `google_map_link: string`
- `image_url: string`
- `qr_code_url: string`
- `owner_id: string` (uid)
- `daily_serial_limit: number`
- `service_minutes_per_vehicle: number` (for ETA)
- `timezone: string` (default `Asia/Dhaka`)
- `current_running_serial: number` (today active pointer)
- `current_date: string` (`YYYY-MM-DD` in local timezone)
- `created_at: Timestamp`
- `updated_at: Timestamp`

### `petrol_pumps/{pumpId}/daily_counters/{YYYY-MM-DD}`
- `next_serial_number: number` (next assignable serial)
- `max_limit: number`
- `booked_count: number`
- `active_count: number`
- `waiting_count: number`
- `done_count: number`
- `expired_count: number`
- `skipped_count: number`
- `running_serial: number`
- `last_reset_at: Timestamp`

### `petrol_pumps/{pumpId}/queue_entries/{entryId}`
- `pump_id: string`
- `user_id: string`
- `serial_number: number`
- `date: string` (`YYYY-MM-DD` local timezone)
- `status: 'waiting' | 'active' | 'done' | 'skipped' | 'expired'`
- `created_at: Timestamp`
- `scanned_at: Timestamp | null`
- `estimated_time: Timestamp`
- `expires_at: Timestamp`

## 5. Queue and Serial Logic

### Serial Generation Rules
- Serial starts from `0` for each pump every local day.
- Max serial count per day equals `daily_serial_limit`.
- Do not allow booking when `next_serial_number >= max_limit`.
- One user can only hold one non-finalized serial per pump per day (`waiting` or `active`).

### Atomic Serial Assignment
Implement as a callable Cloud Function using Firestore transaction:
1. Resolve local date using pump timezone.
2. Read/create `daily_counters/{date}`.
3. Validate limit and duplicate booking constraints.
4. Assign `serial = next_serial_number`.
5. Increment `next_serial_number` and `booked_count`.
6. Create queue entry with computed ETA and expiration window.

### ETA Calculation
- Formula: `(serial_number - running_serial) * service_minutes_per_vehicle`.
- Clamp negative values to 0.
- Show absolute estimated service time and relative waiting minutes.

## 6. QR Check-in Flow

### QR Content Strategy
Each pump QR encodes a URL like:
- `/checkin/{pumpId}?token={shortLivedToken}`

Token options:
- MVP: signed JWT generated by backend and rotated daily
- Alternative: static pumpId only with strong server-side auth checks

### Check-in Process
1. User scans QR and opens check-in route.
2. User must be Google-authenticated.
3. Frontend calls secure function `confirmArrival`.
4. Function verifies:
  - user has today's queue entry at this pump
  - entry status is `waiting`
  - current time is within allowed window
5. Update entry to `active`, set `scanned_at`.

## 7. Auto Expiry and Daily Reset

### Auto Expiry Job (every 1-5 minutes)
- Query queue entries where:
  - `status == waiting`
  - `expires_at < now`
- Mark as `expired` (or `skipped` based on business rule).
- Update aggregate counters.

### Daily Reset Strategy
Avoid mass writes at midnight by using date-scoped counters.
- At runtime, each transaction uses local `YYYY-MM-DD`.
- If no counter exists for new day, create it with:
  - `next_serial_number = 0`
  - `running_serial = 0`
  - `max_limit = pump.daily_serial_limit`

Optional scheduled maintenance at 00:05 local timezone:
- Initialize counters for active pumps.
- Archive previous day metrics.
- Clean stale waiting entries from previous day to `expired`.

## 8. Owner Dashboard Features

### Pump Registration

- On submit:
  - create pump document with slug
  - generate QR code image server-side

### Queue Operations
- View live queue list by status.
- Show current running serial.
- Actions:
  - increment running serial
  - manual set running serial (override with audit trail)
  - mark active entry as done

### Limit Management
- Owner can update `daily_serial_limit`.
- Rule for same-day updates:
  - cannot reduce below `booked_count`.

## 9. Authentication and Authorization

### Auth Flow
- Google sign-in only.
- On first sign-in create `users/{uid}` with default role `user`.
- Allow role upgrade to `pump_owner` after verification (admin process or owner registration approval flow).

### Firestore Security Rules (high level)
- Users can read pump public data.
- Users can read/write only their own queue entries (restricted fields).
- Only callable functions can create serial entries and perform check-ins.
- Owners can write only to pumps they own.
- Owners can read queue entries of their own pumps.
- Prevent direct client edits to protected fields:
  - `serial_number`, `date`, `status` transitions, counters

## 15. Risks and Mitigations
- Race conditions during high booking traffic:
  - use Firestore transactions only via functions
- Timezone mismatch for daily reset:
  - store pump timezone and compute server-side date
- QR misuse or spoofing:
  - signed tokens and auth enforcement
- Owner misuse of manual overrides:
  - store audit log of override actions

## 16. Definition of Done (MVP)
- User can sign in with Google and take serial for a pump.
- Serial respects daily limits and resets each day from 0.
- User can track running serial and ETA in realtime.
- QR check-in marks eligible queue entry as active.
- Missed check-ins auto-expire.
- Owner can manage pump limits and running serial.
- Core pages are mobile-first, fast, and SEO indexed.
